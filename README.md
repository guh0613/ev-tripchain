# ev-tripchain

## 项目总览：ev-tripchain 是什么？

这是一个**电动汽车（EV）配电网承载力评估**的研究框架。用大白话说：

> **一个配电网最多能接入多少辆电动汽车充电，而不导致电网"出问题"（电压过低、线路过载）？**

项目通过 **蒙特卡洛模拟 + 二分搜索** 的方式，找到一个临界值 `N*`——在给定的风险容忍度（比如 5% 的违规概率）下，电网最多能承载的 EV 数量。

## 核心模块详解

### 1. 出行 & 充电建模：`src/ev_tripchain/mobility/`

这是项目最复杂的部分，负责回答：**每辆车什么时候在哪里充多少电？**

项目提供了 **两种建模方式**：

#### 方式 A：简易会话模型（`synthetic.py`）

- 每辆车随机生成若干"充电会话"（开始时间、持续时长、充电功率）
- 类似于"掷骰子"决定什么时候充电，比较简单粗暴
- 配置文件：`configs/example.yaml`

#### 方式 B：出行链 + SOC 模型（更真实）

这是项目的**核心创新点**，由以下几个文件协作完成：

- **`trip_chain.py`**：定义数据结构
  - `Stop`：一次停留（在哪个区域、几点到、几点走、干什么）
  - `TripChain`：一天的出行序列，比如 `家 → 公司 → 商场 → 家`

- **`tripchain_sampling.py`**：随机生成出行链
  - 模拟一个人一天的出行：几点出门、去哪、待多久、开多远
  - 用**对数正态分布**生成行驶距离，**截断正态分布**生成停留时长
  - 保证时间顺序合理（不能还没到就走了）

- **`soc.py`**：电池电量（SOC）逐分钟仿真
  - 开车时电量下降（按 `0.18 kWh/km` 耗电）
  - 到达停留点时，如果电量低于阈值（比如 50%）且该地点可充电，就开始充电
  - 充电直到充满或离开
  - 输出：24 小时的充电功率曲线

- **`spatial.py`**：空间距离模型
  - 内置了 IEEE 33 节点网络的充电站距离表（来自文献）
  - 支持"就近选站"和"导航引导"两种策略

- **`mapping.py`**：把交通区域映射到电网节点
  - 比如"区域 5 的车充电时接在电网的第 12 号母线上"

- **`tripchain_profile.py`**：整合以上模块
  - 对每辆车：生成出行链 → 模拟电量变化 → 应用充电策略 → 映射到电网节点
  - 输出：一个 `T × n_buses` 的矩阵（每个时间步、每个母线上有多少 MW 充电负荷）

#### 四种充电策略

| 策略 | 含义 |
|------|------|
| `uncontrolled` | 无序充电——到了就充，不管别的 |
| `ordered` | 有序充电——只在 22:00-06:00 充电（夜间低谷） |
| `nearest` | 就近充电——去最近的充电站 |
| `navigation` | 导航引导——从最近的 K 个站里随机选一个（分散负荷） |

### 2. 配电网模型：`src/ev_tripchain/grid/`

负责回答：**加了这些充电负荷后，电网还安全吗？**

- **`cases.py`**：加载标准测试网络
  - 支持 `ieee33`（33 节点配电网，最常用）、`cigre_mv`、`4bus`（简单测试用）
  - 可以通过 `load_scale` 调节基础负荷大小

- **`powerflow.py`**：运行交流潮流计算
  - 调用 `pandapower` 的牛顿-拉夫逊法，算出电压、电流、功率分布

- **`constraints.py`**：检查约束是否违规
  - **电压约束**：每个母线电压必须在 [0.95, 1.05] p.u. 之间
  - **线路热约束**：线路负载率不能超过 100%
  - **变压器约束**：变压器负载率不能超过 100%
  - 任何一项超标 → 判定"违规"

---

### 3. 承载力评估：`src/ev_tripchain/hosting_capacity/`

把上面两块串起来，找到临界值 `N*`。

- **`monte_carlo.py`**：蒙特卡洛估计违规概率
  - 跑 S 个场景（比如 50 个），每个场景随机生成 EV 充电负荷
  - 统计有多少场景发生了违规 → 估计违规概率 `π(N)`
  - 用 **Wilson 置信区间** 量化统计不确定性

- **`evaluate.py`**：单次评估的核心逻辑
  - 给定 N 辆车，跑蒙特卡洛，返回违规概率
  - 每个场景内部：生成负荷 → 逐时间步设置负荷 → 跑潮流 → 检查约束

- **`search.py`**：二分搜索找 `N*`
  - 核心思想：`π(N)` 关于 N 是单调递增的（车越多越容易违规）
  - 所以可以用二分法高效找到 `π(N*) ≤ 5%` 的最大 N
  - 大约 14~16 次迭代就能在 [0, 3000] 范围内找到答案

---

### 4. 流水线：`src/ev_tripchain/pipelines/run.py`

把所有模块串成完整的执行流程：

```
读取 YAML 配置
    ↓
加载配电网（如 IEEE 33）
    ↓
定义 risk_at_n(n) 函数：
  "给 n 辆车，违规概率是多少？"
    ↓
二分搜索找 N*：
  反复调用 risk_at_n(n)
  收敛到 π(N*) ≤ ε
    ↓
输出结果 JSON（N* 和风险曲线）
```

---

### 5. 配置系统：`src/ev_tripchain/config.py`

用 **Pydantic** 做严格的配置校验，YAML 配置会被解析成嵌套的 Python 对象：

```yaml
case:
  name: ieee33
  load_scale: 0.55        # 基础负荷缩放

hosting_capacity:
  n_scenarios: 50          # 蒙特卡洛场景数
  risk_tolerance: 0.05     # 5% 风险容忍度
  n_max: 3000              # 搜索上界

mobility:
  model: tripchain_soc     # 选哪种出行模型
  strategy:
    name: uncontrolled      # 充电策略
```

### 6. 分析脚本：`scripts/`

| 脚本 | 功能 |
|------|------|
| `parameter_sweep.py` | 参数扫描：充电功率 × 负荷缩放，看 N* 怎么变 |
| `strategy_comparison.py` | 四种充电策略的 N* 对比 |
| `bottleneck_analysis.py` | 薄弱环节定位：哪个母线电压最低、哪条线路负载最高 |
| `validate_plots.py` | 生成 6 张分析图（出行分布、SOC 曲线、风险曲线等） |

## 关键发现

根据 `docs/weekly-report/report3.md`：

- 出行链模型比简易会话模型得到的 N* **高约 3 倍**（98 vs 34），因为出行链模型的充电时间更分散、更贴近现实
- **瓶颈在 Bus 17**（电压最低，接近 0.95 p.u. 下限），主要约束是电压而非线路过载
- `ordered`（有序充电）策略反而不如无序充电，因为把充电集中到夜间反而造成更高峰值
- `navigation`（导航引导）当前也不如基线，因为随机选站的机制还需要优化

## 总结

这个项目用**蒙特卡洛仿真**模拟大量 EV 的随机充电行为，结合**潮流计算**检查电网安全，通过**二分搜索**找到电网能安全承载的最大 EV 数量，并支持对比不同充电策略的效果。
